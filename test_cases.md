---

## このドキュメントの使い方

1. **事前条件** を整えてからテストを実施する
2. **期待結果** と実際の画面表示・動作を比較する
3. 差分があれば不具合報告書（README §6 フォーマット）に記録する
4. 「INTENTIONAL_BUGS に設定」とある欄は `main.py` の該当行を書き換えて起動する
5. 確率的な挙動（ミス・クリティカル・壁すり抜け）は **最低 20 回以上** 試行してから判断する

> **判断基準**: 「なんかおかしい」ではなく「仕様書（roguelike_spec.md）の記述と違う」で判断する。

## 重点確認ポイント

### テスト実施の優先順位

```
【第1優先】高優先度テスト（53件）を先に通す
  → 基本操作・移動・戦闘・勝敗判定の骨格を確認する

【第2優先】不具合検出テスト（TC-08-01〜06）
  → BUG-01〜06 を1件ずつ有効にして動作確認する

【第3優先】エリアサイズ別確認（TC-07-01〜11）
  → small / large でも基本機能が正しく動くことを確認する
```

### 確率的な挙動への注意点

| 対象 | 確率 | 最低試行回数 | 確認のコツ |
|---|---|---|---|
| ミス（BUG-06 関連） | 10% | 30回以上 | MISS 表示と HP 変動を必ずセットで記録する |
| クリティカル（BUG-03 関連） | ~18% | 30回以上 | 「CRITICAL!(XX)」の数値を毎回メモする |
| 壁すり抜け（BUG-02 関連） | 0%（正常）または 30% | 30回以上 | 壁方向に繰り返し入力して「すり抜け」が起きないか確認する |

> 「1回試して起きなかった = バグがない」とは言えません。確率的なバグは試行回数が少ないと見落とします。

### 仕様書との照合チェックリスト

報告書を書く前に以下を確認する：

- [ ] 期待結果は仕様書の節番号（§XX）を根拠としているか
- [ ] 「おかしいと思う」ではなく「仕様書と異なる」と言えるか
- [ ] 再現手順は「このまま操作すれば他の人でも再現できる」粒度か
- [ ] 確率的なバグは十分な試行回数（20〜30回）を確保したか

### 見落としやすいシナリオ

1. **HP満タン（100）での `H` 使用** → 使用は成立するが回復量は 0（インベントリ -1 は消費される）
2. **ミス後の HP 変動** → MISS 表示のとき HP は絶対に変化しないはず
3. **壁衝突時のターン消費** → ターンが増えていないことを Turn 表示で必ず確認
4. **全敵撃破後のゲーム継続** → 「CLEAR!」が出るのは規定ターン経過後のみ
5. **勝敗の同時成立** → HP=0 かつ Turn=SURVIVE_TURNS のとき GAME OVER が優先
6. **リスタート後の完全初期化** → Score/Turn/Inv/HP がすべて 0/0/0/100 に戻ること
